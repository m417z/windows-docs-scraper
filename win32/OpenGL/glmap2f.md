# glMap2f function

The [**glMap2d**](https://learn.microsoft.com/windows/win32/opengl/glmap2d) and **glMap2f** functions define a two-dimensional evaluator.

## Parameters

*target*

The kind of values that are generated by the evaluator. The following symbolic constants are accepted.

| Value | Meaning |
|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **GL\_MAP2\_VERTEX\_3** | Each control point is three floating-point values representing *x, y,* and *z*. Internal [**glVertex3**](https://learn.microsoft.com/windows/win32/opengl/glvertex-functions) commands are generated when the map is evaluated.<br> |
| **GL\_MAP2\_VERTEX\_4** | Each control point is four floating-point values representing *x, y, z,* and *w*. Internal [**glVertex4**](https://learn.microsoft.com/windows/win32/opengl/glvertex-functions) commands are generated when the map is evaluated.<br> |
| **GL\_MAP2\_INDEX** | Each control point is a single floating-point value representing a color index. Internal [**glIndex**](https://learn.microsoft.com/windows/win32/opengl/glindex-functions) commands are generated when the map is evaluated. The current index is not updated with the value of these **glIndex** commands, however.<br> |
| **GL\_MAP2\_COLOR\_4** | Each control point is four floating-point values representing red, green, blue, and alpha. Internal [**glColor4**](https://learn.microsoft.com/windows/win32/opengl/glcolor-functions) commands are generated when the map is evaluated. The current color is not updated with the value of these **glColor4** commands, however.<br> |
| **GL\_MAP2\_NORMAL** | Each control point is three floating-point values representing the *x, y,* and *z* components of a normal vector. Internal [**glNormal**](https://learn.microsoft.com/windows/win32/opengl/glnormal-functions) commands are generated when the map is evaluated. The current normal is not updated with the value of these **glNormal** commands, however.<br> |
| **GL\_MAP2\_TEXTURE\_COORD\_1** | Each control point is a single floating-point value representing the *s* texture coordinate. Internal [**glTexCoord1**](https://learn.microsoft.com/windows/win32/opengl/gltexcoord-functions) commands are generated when the map is evaluated. The current texture coordinates are not updated with the value of these **glTexCoord** commands, however.<br> |
| **GL\_MAP2\_TEXTURE\_COORD\_2** | Each control point is two floating-point values representing the *s* and *t* texture coordinates. Internal [**glTexCoord2**](https://learn.microsoft.com/windows/win32/opengl/gltexcoord-functions) commands are generated when the map is evaluated. The current texture coordinates are not updated with the value of these **glTexCoord** commands, however.<br> |
| **GL\_MAP2\_TEXTURE\_COORD\_3** | Each control point is three floating-point values representing the *s, t,* and *r* texture coordinates. Internal [**glTexCoord3**](https://learn.microsoft.com/windows/win32/opengl/gltexcoord-functions) commands are generated when the map is evaluated. The current texture coordinates are not updated with the value of these **glTexCoord** commands, however.<br> |
| **GL\_MAP2\_TEXTURE\_COORD\_4** | Each control point is four floating-point values representing the *s, t, r,* and *q* texture coordinates. Internal [**glTexCoord4**](https://learn.microsoft.com/windows/win32/opengl/gltexcoord-functions) commands are generated when the map is evaluated. The current texture coordinates are not updated with the value of these **glTexCoord** commands, however.<br> |

*u1*

A linear mapping of *u*, as presented to [**glEvalCoord2**](https://learn.microsoft.com/windows/win32/opengl/glevalcoord-functions), to *u*^, one of the two variables that is evaluated by the equations specified by this command.

*u2*

A linear mapping of *u*, as presented to [**glEvalCoord2**](https://learn.microsoft.com/windows/win32/opengl/glevalcoord-functions), to *u*^, one of the two variables that is evaluated by the equations specified by this command.

*ustride*

The number of floats or doubles between the beginning of control point **R** *ij* and the beginning of control point **R** (i\ +1\ )\ j, where *i* and *j* are the *u* and *v* control point indexes, respectively. This allows control points to be embedded in arbitrary data structures. The only constraint is that the values for a particular control point must occupy contiguous memory locations.

*uorder*

The dimension of the control point array in the *u*-axis. Must be positive.

*v1*

A linear mapping of *v*, as presented to [**glEvalCoord2**](https://learn.microsoft.com/windows/win32/opengl/glevalcoord-functions), to *v*^, one of the two variables that is evaluated by the equations specified by this command.

*v2*

A linear mapping of *v*, as presented to [**glEvalCoord2**](https://learn.microsoft.com/windows/win32/opengl/glevalcoord-functions), to *v*^, one of the two variables that is evaluated by the equations specified by this command.

*vstride*

The number of floats or doubles between the beginning of control point **R** *ij* and the beginning of control point **R** i(j\ +1\ ), where *i* and *j* are the *u* and *v* control point indexes, respectively. This allows control points to be embedded in arbitrary data structures. The only constraint is that the values for a particular control point must occupy contiguous memory locations.

*vorder*

The dimension of the control point array in the *v*-axis. Must be positive.

*points*

A pointer to the array of control points.

## Return value

This function does not return a value.

## Error codes

The following error codes can be retrieved by the [**glGetError**](https://learn.microsoft.com/windows/win32/opengl/glgeterror) function.

| Name | Meaning |
|-------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------|
| **GL\_INVALID\_ENUM** | *target* was not an accepted value.<br> |
| **GL\_INVALID\_VALUE** | *u1* was equal to *u2*, or *v1* was equal to *v2*.<br> |
| **GL\_INVALID\_VALUE** | Either *ustride* or *vstride* was less than the number of values in a control point.<br> |
| **GL\_INVALID\_VALUE** | Either *uorder* or *vorder* was less than one or GL\_MAX\_EVAL\_ORDER.<br> |
| **GL\_INVALID\_OPERATION** | The function was called between a call to [**glBegin**](https://learn.microsoft.com/windows/win32/opengl/glbegin) and the corresponding call to [**glEnd**](https://learn.microsoft.com/windows/win32/opengl/glend).<br> |

## Remarks

Evaluators provide a way to use polynomial or rational polynomial mapping to produce vertices, normals, texture coordinates, and colors. The values produced by an evaluator are sent on to further stages of OpenGL processing just as if they had been presented using [**glVertex**](https://learn.microsoft.com/windows/win32/opengl/glvertex-functions), [**glNormal**](https://learn.microsoft.com/windows/win32/opengl/glnormal-functions), [**glTexCoord**](https://learn.microsoft.com/windows/win32/opengl/gltexcoord-functions), and [**glColor**](https://learn.microsoft.com/windows/win32/opengl/glcolor-functions) commands, except that the generated values do not update the current normal, texture coordinates, or color.

All polynomial or rational polynomial splines of any degree (up to the maximum degree supported by the OpenGL implementation) can be described using evaluators. These include almost all surfaces used in computer graphics, including B-spline surfaces, NURBS surfaces, Bezier surfaces, and so on.

The **glMap2** function is used to define the basis and to specify what kind of values are produced. Once defined, a map can be enabled and disabled by calling [**glEnable**](https://learn.microsoft.com/windows/win32/opengl/glenable) and **glDisable** with the map name, one of the nine predefined values for *target*, described above. When [**glEvalCoord2**](https://learn.microsoft.com/windows/win32/opengl/glevalcoord-functions) presents values *u* and *v*, the bivariate Bernstein polynomials are evaluated using *u*^ and *v*^, where

![Equation showing the definition of u^.](https://learn.microsoft.com/windows/win32/opengl/images/map09.png)

and

![Equation showing the definition of v^.](https://learn.microsoft.com/windows/win32/opengl/images/map10.png)

The *target* parameter is a symbolic constant that indicates what kind of control points are provided in *points*, and what output is generated when the map is evaluated.

The *ustride*, *uorder*, *vstride*, *vorder*, and *points* parameters define the array addressing for accessing the control points. The *points* parameter is the location of the first control point, which occupies one, two, three, or four contiguous memory locations, depending on which map is being defined. There are *uorder* x *vorder* control points in the array. The *ustride* parameter tells how many float or double locations are skipped to advance the internal memory pointer from control point **R** *ij* to control point **R** (\ i+1\ )j. The *vstride* parameter tells how many float or double locations are skipped to advance the internal memory pointer from control point **R** *ij* to control point **R**i(j\ +1\ ).

As is the case with all OpenGL commands that accept pointers to data, it is as if the contents of *points* were copied by **glMap2** before it returned. Changes to the contents of *points* have no effect after **glMap2** is called.
